# 合约详情页完善计划

## 一、目标

把当前占位性质的合约详情页改造成单币种分析入口，回答四个问题：

1. 当前价格、资金费率和持仓处于什么状态。
2. 价格、持仓量和主力净流入在各周期如何变化。
3. 各交易所的数据是否一致，哪家交易所贡献最大。
4. 当前数据是否新鲜、完整，能否用于判断。

保持单体架构和一条命令启动：Flask、Jinja、Vue 3 CDN、ECharts、SQLAlchemy，不增加前端构建和新服务。

## 二、当前问题

- 页面中的周期持仓变化是 `intervalChanges` 硬编码示例，并非真实数据。
- 页面读取 `net_inflow_data`，但 `/api/coin-detail/<symbol>` 没有返回该字段。
- 详情接口在每次请求中调用 Binance 实时接口，响应速度和稳定性受外部网络影响。
- 已落库的多交易所 K 线、持仓、主动买卖量和结构评分没有在详情页复用。
- `long_short_ratio` 已由接口返回，但页面没有展示。
- 缺少更新时间、数据状态、缺失交易所等可信度信息。
- 只有数值卡片，没有历史趋势，无法判断指标是持续变化还是瞬时波动。
- 默认币种写死为 `SPKUSDT`；非法或不存在的 symbol 没有明确的 400/404 语义。
- Playwright 只验证少量静态文本，后端详情接口没有独立契约测试。

## 三、页面信息架构

### 3.1 顶部栏

- 合约名称、跟踪状态、数据状态。
- 最近更新时间和距今时长。
- 返回按钮、刷新按钮。
- 上一个/下一个币种暂不做，避免引入额外状态。

### 3.2 核心摘要

- 最新价、24h 涨跌、24h 成交额。
- 总持仓价值、5m/1h/4h/24h 持仓变化率。
- 当前/预测资金费率、下次结算倒计时。
- 市场结构总分、方向和置信状态；无评分时显示暂无数据。

### 3.3 趋势图

使用一个共享时间范围切换器：`1h / 4h / 24h / 72h / 7d`。

- 价格与持仓价值双轴图。
- 主动买入、主动卖出和净流入图。
- 资金费率历史图。

第一版不做 K 线蜡烛图，折线图已经能回答趋势问题，也更容易维护。

### 3.4 多周期矩阵

复用首页已经稳定的周期口径，展示：

- 周期：5m、15m、30m、1h、4h、12h、24h、48h、72h、168h。
- 价格变化率。
- 持仓数量变化率和持仓价值变化率。
- 累计净流入金额。

### 3.5 交易所对比与数据质量

- Binance、OKX、Bybit、Gate 的持仓量、持仓价值和占比。
- 各交易所最后数据时间。
- `included_exchanges`、`missing_exchanges` 和整体状态。
- 部分缺失时保留可用数据，不让整个页面失败。

## 四、API 设计

保留入口：

```text
GET /api/coin-detail/<symbol>?range=24h
```

建议响应分区，避免继续使用零散顶层字段：

```json
{
  "status": "success",
  "data": {
    "symbol": "BTCUSDT",
    "as_of": 1780000000000,
    "data_status": "complete",
    "missing_exchanges": [],
    "summary": {},
    "intervals": [],
    "exchange_distribution": [],
    "charts": {
      "market": [],
      "flow": [],
      "funding_rate": []
    },
    "structure_score": null
  }
}
```

约束：

- symbol 统一转大写，仅允许字母、数字及常见合约分隔符，非法输入返回 400。
- 数据库中不存在且交易对配置中不存在时返回 404。
- `range` 使用白名单，非法值返回 400。
- 数值字段返回数字或 `null`，格式化留给前端；时间统一使用毫秒时间戳。
- 数据部分缺失仍返回 200，并通过状态和缺失字段表达。
- API 默认只读数据库，不在页面请求中同步调用交易所。

## 五、实现阶段

### 阶段 0：修正现有详情契约

- [x] 新增详情 API 测试，锁定 symbol 校验、404、部分数据和异常行为。
- [x] 删除前端硬编码 `intervalChanges`。
- [x] 修正 `net_inflow_data` 字段不匹配。
- [x] 取消默认 `SPKUSDT`，缺少 symbol 时显示明确错误或从跟踪列表选择。
- [x] 将 API 数据聚合放入新仓库模块 `repositories/contract_detail.py`。

验收：页面展示的每个业务数值都来自 API；断网但数据库有数据时仍可打开详情。

### 阶段 1：可用的详情页 MVP

- [x] 复用 `get_homepage_series_snapshot([symbol])` 生成摘要、多周期矩阵和交易所分布。
- [x] 复用资金费率仓库生成当前费率和预测费率；历史曲线留在阶段 2。
- [x] 复用市场结构评分仓库返回该币种最新评分。
- [x] 增加更新时间、完整/部分缺失/无数据状态。
- [x] 重做详情页布局，补齐加载、空数据、部分失败和重试状态。
- [x] 从首页、行情榜、结构评分和资金费率页统一链接到详情页。

验收：用户可以从主要列表进入详情，并在一个页面看到当前摘要、周期变化、交易所分布和评分。

### 阶段 2：趋势图

- [x] 在详情仓库增加按 symbol、exchange、时间范围批量读取序列的方法。
- [x] 返回统一时间轴的价格、持仓价值、买卖量和净流入序列。
- [x] 使用 ECharts 实现市场、资金流和资金费率三张图。
- [x] 支持时间范围切换，图表请求加加载和空数据状态。
- [x] 限制最大数据点数，长周期在仓库层抽样，避免返回超大响应。

验收：切换范围不会触发交易所实时请求；24h 页面响应和图表数据量保持可控。

### 阶段 3：增强项，按实际使用反馈决定

- [ ] 自选与备注。
- [ ] 为当前合约创建预警规则。
- [ ] 异常事件时间线。
- [ ] 多交易所资金费率/价格差与套利估算。
- [ ] 指标截图或 CSV 导出。

这些功能不进入 MVP，避免详情页项目扩大成预警或策略系统重构。

## 七、测试清单

- 合法/非法/不存在的 symbol。
- 完整数据、单交易所缺失、资金费率缺失、全部无数据。
- 周期变化和净流入计算口径与首页一致。
- 时间范围边界、排序、最大点数和时间戳对齐。
- 页面直达、从其他页面进入、刷新和返回。
- API 失败、空图表和窗口缩放时的图表行为。
- 移动端与桌面端无内容重叠。

## 八、推荐执行顺序

先完成阶段 0 和阶段 1，作为一个版本交付；稳定使用后再做阶段 2。阶段 0/1 主要复用现有仓库，风险低且能立即消除假数据。阶段 2 涉及序列查询、对齐和图表，单独开发与验证更容易控制范围。
